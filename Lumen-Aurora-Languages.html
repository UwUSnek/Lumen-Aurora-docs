Lumen and Aurora are high-level programming languages designed to work together seamlessly, offering an integrated solution for both general-purpose programming and GPU computing.

Lumen
Programming language
Based on LLVM
Aurora
Shading language
Based on SPIR-V

Modern languages often trade runtime performance for simplicity, relying on the false assumption that whatever machine will be running the code is powerful enough to render the overhead negligible.
This creates a system where inefficiencies keep piling up, resulting in sub-optimal performance, even in the simplest of programs.

Furthermore, the combination of runtime data and complex semantics makes the debugging process needlessly time consuming.
This is often accompanied by an unreadable syntax and less-than-optimal error detection and reporting systems, all of which exacerbate the problem.

Lumen and Aurora exist to address these issues.
A shared static type system confines type-related issues to the compilation phase.
Simple semantics and a modular code structure simplify the debugging process and make it easier to keep a mental model of the program.
Templates and Directives allow for expressive but elegant syntax, and heavy optimization ensures maximum runtime performance.

Lumen is a general-purpose high-level language with a C++-like syntax.
Its purpose is to provide the speed and power of low-level languages, without sacrificing readability and maintainability.
The language achieves this by offering features such as modules, reflection, a powerful optimizer and native support for both Windows and Linux.

Lumen's optimizer focuses entirely on maximizing performance.
Before any optimization happens, the code is thoroughly parsed and simplified, so the optimizer can work without compromises.
This ensures that the performance is on par with C, while keeping the program's behaviour exactly as intended.
Advanced features like templates and reflection come with no performance penalty, letting developers focus on writing clean and effective code.

Lumen includes built-in checks for common issues like overflows and race conditions, and allows developers to add custom ones.
They are easily toggleable and are meant for use during development.
This ensures that developers can catch problems before they escalate, making the code more robust without affecting performance when deploying the final product.

Unlike most modern languages, Lumen is not Object-Oriented.
Instead, features usually associated with OOP are covered by templates, reflection and the language's syntax.
These provide better runtime performance while maintaining flexibility.
For a complete list of omitted features, see Omitted features.

Traditionally, the preprocessor is a separate program invoked before compilation. It is responsible for splicing lines, removing comments, and executing preprocessor directives.
The ALC compiler implements the preprocessor as a compilation phase. This lets it generate better diagnostics and reduce overhead.

This phase is fully independent and works regardless of the languages' lexicon and semantics.
At this stage, tokens haven't been formed yet. The entire source code is treated as a stream of individual characters.

    Preprocessor operations are performed in the following order:
    • 1. Include phase.
      Include statements are recursively replaced with the contents of the specified file.
    • 2. Line splicing.
      Line splicing tokens are removed, merging affected lines.
    • 3. Cleanup phase.
      All comments are removed.

NOTICE:
Splitting lines can easily lead to unreadable code. Proper language syntax should be preferred whenever possible.
Line splicing tokens are provided purely for completeness and familiarity. Use them responsibly.

Line splicing can be used to merge multiple lines and allow comments, include statements and language tokens to be written across multiple lines.

    The preprocessor supports two types of line splicing tokens:
    • Line continuation token: It consists of a \ immediately followed by a character.
      This type of line splicing token doesn't trim any whitespace. All adjacent tabs and spaces are preserved. Only the token itself (which includes the character) is removed.
    • Line junction token: It consists of two \ characters immediately followed by a character.
      This type of line splicing token trims any whitespace before it and at the start of the next line. This allows it to remove code indentation.

Line splicing tokens cannot be nested.
Using a line splicing token at the end of a file without a trailing is allowed and has no effect.

Comments are portions of code that are intentionally ignored.
They can be used to provide information or temporarily exclude lines of code.
Comments are allowed to contain any sequence of Unicode characters of any length.

    The preprocessor supports two types of comments:
    • Single line comments start with // and end immediately before the next character (\n).
    • Multiline comments start with /* and end with */.

Comments cannot be nested.
Unclosed comments are allowed. They include everything from their opening character sequence to the end of the file.

Just like line splicing tokens, they interact uniquely with include statements, as the include statement parser specifically skips comments, making it possible to place them anywhere within it, except inside the file path, which is a text literal.

Comments are removed after the line splicing phase. This means that line splicing tokens are allowed and work normally within comments.

Unclosed comments cannot bleed into other files. They always end where their file ends.
Including a file containing an unclosed comment is safe and doesn't cause the rest of the code to become commented.

Because the character at the end of single-line comments is not considered part of the comment, it is not removed by the preprocessor.

The contents of string and character literals are excluded from the cleanup phase (comment removal).
This is done in order to allow literals to contain any sequence of characters, including comment-like patterns, without the need for special syntax.
Internally, the preprocessor tries to form text literal tokens using the same logic as the compiler's tokenizer. Valid tokens are skipped entirely, resuming comment removal from the first character after their closing delimiter.
This ensures that any valid Lumen or Aurora text literal is kept unaltered during the cleanup phase.

NOTICE:
The languages provide include statements, but they should be avoided whenever possible.
The import directive does a much better job at importing other modules, looks cleaner and can be debugged easily.
Include statements should only be used if you know what you are doing and you are certain that import directives won't be enough.

The include statement is used to paste code from other files.
Any existing file with read permissions can be included, regardless of its contents or extension.

An include statement consists of the sequence of characters #include followed by a string literal containing the path to the target file.
Any amount of whitespace may be present between the #include and the file path.
Standard modules can be included. They are accessed via the reserved ./standard path.
Because the path is a standard string literal, all escape sequences are available within it.

Absolute and relative paths are both supported. Relative paths are relative to the file the include statement is used in or any one of the specified include paths.
Glob patterns are also supported.

Include operations are performed by replacing the include statement (from the # character to the string's closing ") with the contents of the included file.
Every byte of the source file is copied without modification, making include statements recursive.

    File paths have some limitations:
    • Files cannot be named . or ... . identifies the current directory, while .. identifies the parent directory.
    • Local directories called standard are not accessible. standard is a reserved name for the directory containing all standard modules.
    • File names cannot contain any of the following characters: *?\. These are reserved for glob pattern wildcards. Backslashes are not valid on Windows systems. Any other character is allowed.
    • ~ is not recognized as the home directory but as the literal character ~.
    • / identifies the root directory.
    When a glob pattern is used, all the source files are copied in the same way, one after the other, without adding separators in between.
    Matched files are copied in alphabetical order based on their absolute path.
    Glob patterns that don't match any file are allowed and have no effect.
    The preprocessor supports these glob pattern wildcards:
    •   ** - Any sequence of characters including / (any length, including 0).
    •   * - Any sequence of characters but / (any length, including 0).
    •   ? - Any character but /.

The last variable-length wildcard (* or **) of a glob pattern stops before the last occurrence of the character or wildcard that follows them.
Variable-length wildcards that are not last, instead, stop before the very first occurrence of the character or wildcard that follows them.
They are evaluated from left to right.

Glob patterns never match standard modules.

Symlinks in the included path are always resolved.

Syntactic elements are sequences of one or more tokens or other syntactic elements that follow specific patterns.
They define the contents and interface of the module.

Directives are special instructions able to modify, generate, and test source code. They are also used to define how different modules interact with each other.

Declarations define user-defined elements, such as structs, enums, routines, namespaces, templates, symbol aliases and variables.
Anything defined through a declaration is called a Symbol. Symbols can be referenced through their path. This is explained in Name resolution.
There are a total of 11 user-defined elements. They are available in both languages.

Statements perform actions during runtime. They define the behaviour of each routine, their return value and runtime performance.
Statements are hard coded into the languages and cannot be modified. There are a total of 16 statements, 2 of which can only be used in their respective language.

Expressions generate values. They are mostly used to declare symbols and call routines.

Paths are used to reference things declared in the code, either explicitly or implicitly, in order to retrieve or pass information about them.
Any accessible module, namespace, symbol alias, primitive type, enum, struct, special type, routine, variable and temporary variable is considered a symbol.
Paths are a foundamental part of most semantic elements.

Type paths are a subset of Symbol paths that point to symbols suitable to be used as types, such as primitive types, enums, structs and special types.

Reflection paths are a subset of Symbol paths that can access compile time information about the referenced symbol. These are detailedly explained in Reflection.

Scopes define the path to the symbols declared within them.
Any portion of code enclosed in {} tokens defines a new scope. The only exception is the module scope, which contains anything declared in the module.

User defined elements, or Symbols, can be referenced to use them as types, access their value or perform routine calls. This is done through their name or path.

The name is the identifier that was used to declare the symbol, optionally followed, if the symbol is a template, by <> tokens containing the types that identify the generated symbol.
Names are used to reference symbols that are declared in the current scope or in one of the enclosing ones.

Paths, instead, are used to reference symbols declared in scopes that are not parents of the current scope.
They consist of the names of the enclosing scopes of the symbol and the name of the symbol itself separated by . tokens, starting from the outermost.
The . token is used to access the children symbols of the scope.
Paths work in the same way as file paths, with scopes acting like directories.

The path of a symbol can start from any of its enclosing scopes.
Symbols declared within unnamed scopes cannot be referenced from the outside.
The module identifier is used to reference the scope of the current module.
A path that starts from the module scope of the current module or any of the imported ones is called an absolute path.

When the identifier is used, the symbol declared in the innermost scope, relatively to where the identifier is used, always takes precedence. This can be avoided by starting the path from a parent scope so that the identifier is not ambiguous.
Any identifier can be shadowed, including primitive types, the module identifier and symbols from the core module.

"Value" refers to any information stored in a typed container during runtime.
These include Variables, Routine parameters, Enum elements, Temporary variables, and the result of member access and pointer dereferencing on non-temporary values.

Each value has a defined lifetime that starts when it's created and ends when execution reaches the end of its enclosing scope.

The process of populating a typed container with data for the first time is called initialization. All containers must be initialized inside of their declaration. This ensures uninitialized data is never accessible.

    The way containers are initialized is defined by their type:
    • Struct types:
      All members are initialized in order of declaration using their initializer value, then one of its constructors is called based on the provided arguments.
      If no arguments are provided, the constructors that takes 0 arguments is called.
    • Primitive types and pointer types:
      The value is copied from the provided initializer value.
      If no initializer value is provided, all of their bytes are initialized to 0.
    • Enum types:
      The value is copied from the provided initializer value.
      Not providing initializer values is forbidden.

Data can be copied from and to containers for a variety of reasons. This is usually done when passing parameters to routines or assigning values to existing containers.

By default, copying a value involves copying each byte of its binary representation from the source value to the destination container. This is a shallow copy and doesn't duplicate the data referenced by pointer values.
Struct types can declare copy functions to define custom copying logic.

The process of cleaning up the data stored in a typed container is called destruction.
This is done when the container reaches the end of its lifetime.

By default, this doesn't involve any operation other than freeing memory, which is done automatically by the compiler.
Struct types can declare drop functions to define custom deinitialization logic.
Drop functions of struct members are called after the parent struct's drop function.

Constant expressions are expressions that can be computed during the compilation phase.
They can be used within compile-time contexts such as specialization constraints and as initializer values of const declarations.

The concept of a "constant expression" has a much broader scope than in most languages, but it follows very strict rules and doesn't include everything that could theoretically be computed during the compilation phase.
This is to simplify semantics and minimize the amount of rules and exceptions one has to take into account when using them.

    Whether an expression is constant is determined in the following way:
    • A routine is suitable for constant expression calls if it is not declared with the :noconst annotation and all expressions within its signature and body, including implicit init, copy and drop calls, are constant.
    • A routine call is constant if the routine is suitable for constant expression calls and all of its arguments are constant expressions.
    • A conditional value is constant if its condition and the selected value are both constant expressions.
    • A variable, temporary variable or routine parameter is constant if declared with the const qualifier.
    • Literals, enum elements, reflection data, function values, and the result of try expressions are always constant.
Example
import io; // Suitable for constant calls ulong increment(ulong n) { return n + 1; } // Not suitable for constant calls ulong incrementAndLog(ulong n) { io.println("Incrementing value"); return increment(n); } void main() { const int n = increment(5); // Ok, increment call is constant. can be used as initializer // const int n = incrementAndLog(5); Error, incrementAndLog is not suitable for constant calls as it depends on io.println which is :noconst }

Constant expressions are not required to run during the compilation phase, and they don't affect optimizations in any way. Their purpose is solely to be able to be used in contexts where a constant value is required.

Any routine declared in standard modules that depends on runtime data is properly marked as :runtime.

Certain actions cause undefined behaviour.
Undefined behaviour means that the code compiles with no errors, but the runtime behaviour is unknown and anything could happen.
The software could break, have bugs or work just fine. One cannot know.

As with undefined values, the module is responsible for preventing or documenting undefined behaviours.

This section details the functioning and usage of each component of the languages, providing examples for common use cases.

A module is a file containing Lumen or Aurora source code. Modules are used to organize the program into self-contained units.

By default, symbols declared in a module are not accessible from the outside. Modules can define which symbols are available to other modules through the export directive. These symbols can then be accessed using the import directive.

Each module can define a main function. This is where execution begins when the program is started, but only if the module was passed to the compiler as the entry point. This is covered in detail in Compilation.

The global initialization order defines the order in which variables declared outside of routines are initialized. This only affects global variables with an initializer value and enum elements.

Global variables and enum elements of a module are initialized in the same order in which they are declared.
The order in which modules are initialized depends on the import graph of the program: each module is guaranteed to be initialized before any module that depends on it.

The initialization order is only defined when there are no circular dependencies in the import graph.
Circular module dependencies cause all modules in the cycle to have no defined initialization order relative to each other, though they are still guaranteed to be initialized before modules that depend on them but are not in the cycle.

Circular dependencies between initializer values are forbidden.

Global variables are initialized by a special __init function which cannot be called directly. This is where execution actually begins. After all global variables and enum elements have been initialized, the main function is called.

const variables and enum elements are technically initialized during the compilation phase and often optimized out completely, albeit without changing the observable behaviour of the program.
This means that const values are always available to non-const expressions, but for the purposes of initialization, they behave as if initialized during the global initialization stage. This ensures the initialization rules are kept simple and consistent.

All source code that makes up a program is contained within the root scope. The root scope can be thought of as an unnamed namespace. The current module is equivalent to an unnamed namespace inside the root scope.

Imported modules also reside in the root scope. They are equivalent to named namespaces containing the imported symbols. The __internal module is always imported automatically, as it contains the primitive types and fundamental functions.

Full program structure

Directives are special compile-time instructions that modify, generate, and test source code. They also control how different modules interact with each other.

Each directive has its own syntax and behaviour, which are explained in their dedicated sections.

The Import directive allows the current module to access and rename symbols declared in other modules.
Only exported symbols can be imported.

The Import directive has two forms: import, which imports all exported symbols from a module, and from import, which imports only the specified symbols.
Both the module and the imported symbols can be aliased using the as keyword.

When an Import directive is used, it creates a new namespace in the root scope containing the imported symbols.
Anything these symbols depend on is not accessible unless explicitly imported.

The new namespace is named after the name of its file without extensions, or, if specified, the identifier used in the import directive.
An explicit identifier is required when the file name is not a valid identifier.

The Export directive allows a module to define which symbols other modules are allowed to use when importing from it.

The Export directive can also be used to define which members of a struct type can be accessed from outside the struct. This is explained in detail in Structs.

Enums are types whose allowed values are a named subset of the possible values of their base type. The allowed values are called Enum elements.
Unlike most languages, the base type can be any type, including pointers, structs, other enums or even void.

Enum elements work similarly to struct values, containing their name, index, and base value as members. They are part of the global initialization order and their base value is initialized using one of the base type's init functions.
If the base type defines a drop function, it is called on each element before the process ends.

Expressions of enum type are always 8 bytes in size. They are implicitly convertible to any type their base type is implicitly convertible to.
This is explained in detail in Implicit conversions.

Example
import io; enum TempThreshold : float { AbsoluteZero = -273.15; Freezing = 0; Boiling = 100; } void main() { io.println("Water boils at " + TempThreshold.Boiling.value + "°C"); }
Output
Water boils at 100°C

Enum elements are equivalent to global const variables, but enum values simply reference the existing elements. This means that creating or destroying enum values doesn't call any init or drop function of the base type.

An enum is declared using the enum keyword followed by its name and base type. Enum elements are declared in the same way as variables, but the type is implicit and their initializer values must be constant expressions of the base type.

Elements of integral enums with no initializer value are given the value of the preceding element + 1, or 0 if they are the first element.
Elements of void enums cannot be initialized explicitly.
Elements of enums of any other base type must always be initialized.

Multiple elements can have identical values, but each identifier can only be declared once.
Empty enums are not allowed.

Structs are types composed by multiple members that can have differing type.
They are especially useful for complex or non-uniform data that cannot be properly represented using primitive types.

Struct values can be copied and passed as a whole, just like values of primitive type, but they maintain their internal layout. Members can be accessed individually using the . keyword.

The size of a struct type is typically greater than the sum of sizes of the individual members. This is because of alignment requirements adding unused padding space between members. This doesn't affect the language in any way, but it might change how the data is represented in binary formats.

Empty structs are 1 byte in size.

The compiler is allowed to freely reorder struct members or pack them by removing padding space in order to maximize runtime computational speed.
The compiler annotations :nopacking and :noreorder can be used to avoid this.

A struct is declared using the struct keyword followed by its name.
Empty structs are allowed.

Struct members are declared in the same way as variables. They are initialized when new values of the struct type are created, before the init function or any other routine can access them.
Struct members can be of any type, including other structs.

The init function is a special function used to create values of struct type.
It has no name and no explicit return type and it's declared directly inside of the struct's body using the init meta keyword.
Only one init function is ever called for each instance of a struct.
Init functions cannot be called after the value has been created.

By default, each struct contains one implicit init function that takes an argument for each member. Members are initialized with their corresponding argument in order of declaration.
The implicit init function is not present in structs that define at least one explicit init function.

Explicit init functions must initialize all members of the struct.
Init functions are allowed to take no arguments.
There is no limit to the number of init functions a struct can define.

Routine specialization allows routines to act differently based on the numer of arguments in each routine call, their types and user-defined constraints.
This is achieved by declaring the same routine multiple times. Each declaration must be unique and can differ by:

    • Number of parameters (and position of the parameters for operators)
    • Types of the parameters
    • Suitability constraints

Declarations that have less than #a parameters or more than #a parameters without a default value (and for operators, those whose position of the parameters don't match with the position of the arguments in the operator call) are discarded.
[with #a = number of passed arguments]

The remaining declarations are organized in categories and the best one is chosen from one of these.
The categories are checked one at a time, in the order a, b, c. If no declaration is found, the next category is checked.
If zero or more than one declaration are found withing the same category, a semantic error is generated.

  1. Declarations that can take all the parameters without performing implicit conversions.
  2. Declarations that can take all the parameters by performing one or more constness-only implicit conversions.
  3. Declarations that can take all the parameters by performing one or more implicit conversions of any kind.
 Example 1

abcc

Declarations with parameter packs and/or templates undergo an additional instantiation phase before the best declaration is chosen, in which these features are replaced with standard types and parameters.

  1. Each declaration that uses a function parameter pack and has #a or less parameters has its parameter pack replaced with the exact number of parameters of the correct type in order to bring the total to exactly #a.
    [with #a = number of passed arguments].
    This phase is skipped if the template arguments are specified explicitly.
  2. Generated declarations whose number of arguments and argument types are identical to the ones in declarations that don't use function parameter packs are discarded.
  3. Each declaration that uses a template is instantiated.
    This phase is skipped if the template arguments are specified explicitly.
  4. Generated declarations whose number of arguments and argument types are identical to the ones in declarations that don't use templates are discarded
  5. Any declaration whose suitability constraint is not satisfied is discarded

Arithmetic operators can be used on any primitive type or enum.

++, --.
Increment and decrement operators add and subtract 1 to the value of the operand.
Prefix operators return the value of the operand after the operation, postfix operators return the value of the operand before the operation.
Incrementing or decrementing a bool has the same effect of converting its value to int, performing the operation and converting it back to bool.
Increment and decrement are the only common operators which modify the original value.

-, +.
The promotion operator has no effect and only exists for completeness.
The inversion operator inverts the sign of the operand.
Inverting a bool has no effect.

+, -, *, /, %.
Addition, subtraction, multiplication, division and modulus operators perform their corresponding operations between two operators.
Using these operators with bool values has the same effect of converting their value to int, performing the operation and converting them back to bool.

Differently from GLSL and C++, the result of the modulus operator has the same sign of the result of the division.
Using 0 as second operand of in division or modulus operator with integral types has undefined result.

Logical operators can only be used on bool types.

!, &&, ||, ^^.

Bitwise operators can be used on any primitive type or enum. //TODO convert floats to int without changing the bit value //TODO floatBitsToInt intBitsToFloat

<<, >>.
Bit shift operators shift the bits of the value of the left operand by the value of the right operand.
The right operand is converted to uint. Discarded bits are set to 0.
Shifting non-bool types by sizeof(type) * 8 or more causes undefined behaviour.
Shifting bool values is equivalent to setting them to false.

~, &, |, ^.
The bitwise NOT performs the logical NOT operation on every bit of the operand.
Bitwise AND, OR and XOR operators are equivalent to their corresponding logical operators, but the operation is performed between each bit of the two operands.
The types of the operand must have the same size.
Implicit conversions don't apply on these operators. The return type has the same type of the first opeand. //TODO this is different than c++ and glsl

Relational operators can be used on any primitive type or enum, with the exception of ==, !=, =: and !: which can be used with any type, including structs and typenames.

==, !=, <, >, <=, >=.
The equality operator returns true if the operands have the same value.
The inequality operator returns true if the operands have different values.
The other operators return true if the first operand is less than, greater than, less or equal than or greater or equal than the second operand, respectively.

=:, !:.
The existence operator returns true if the first operand has the value of one of the elements of the second operand.
The inexistence operator returns true if all the elements of the second operand are different than the value of the first operand.
The second operand must be an array with a base type that is the same as the type of the first operand or can be implicitly converted to it.

Their return value is the same as if they were declared in the following way:

Existence
bool (baseof(t) a) =: (?t b) { for(uint i = 0; i < b.len(); ++i) { if(a == b[i]) return true; } return false; }
Inexistence

Assignment operators are used to change the value of l-value symbols.
The original value is set to the result of the operation.

The direct assignment (=) changes the value of the first operand to the one of the second operand.
It can be used with type, including typenames and composite types. Structs must be of the same exact type to be assigned.

The result of the other assignment operators and the types they can be used with are the same as their corresponding common operators.

All common and assignment operators in the core module, excluding =: and !:, can be used to operate on whole arrays.

When arrays are used with unary operators, the operation is performed on all the elements of the array and the return value is an array of the same length, containing the results of the operations.

When arrays are used as both the first and second operand of binary operators, the arrays must have identical length. The operation is performed between each element of the first operand and the respective element of the second operand, and the return value is an array of the same length containing the results of the operations.

When an array is used as the first operand of binary operators and a value of its base type is used as the second operand, the operation is performed between each element of the first operand and the value of second operand. The return value is an array of the same length of the first operand containing the results of the operations.
