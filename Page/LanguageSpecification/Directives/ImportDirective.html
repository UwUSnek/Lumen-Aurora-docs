The Import directive allows the current module to access and rename symbols declared in other modules.
Only exported symbols can be imported.

Both the module and the imported symbols can be aliased using the as keyword. Anything the imported symbols depend on is not accessible unless explicitly imported.

    The Import directive has four forms:
    • import: imports all exported symbols from a module
    • from import: imports only the specified symbols
    • include: imports all exported symbols from a module without wrapping them in a namespace
    • from include: imports only the specified symbols without wrapping them in a namespace

When either of the import forms is used, it creates a new namespace in the root scope containing the imported symbols.
The new namespace is named after the name of its file without extensions, or, if specified, the identifier used in the import directive.
An explicit identifier is required when the file name is not a valid identifier.
Aliasing the module while using include is allowed and has no effect.

Import syntax Module interface name Module interface element Example 1 - test.lmn
str message = "Hello :3"; export;
Example 1 - main.lmn
include "./test.lmn"; import io; void main(){ io.print(message); }
Output
Hello :3
Example 2
from math import sum; from io as console import print as write; void main(){ console.write(math.sum(8, 9, 10)); }
Program structure
// Root scope namespace { namespace math { // sum function... } namespace console { // print function ... [renamed to "write"] } // Module scope namespace { void main(){ console.write(math.sum(8, 9, 10)); } } }

Modules are always preprocessed and partially compiled before being imported.

Using an import directive flags the imported module as a dependency of the current module. This has the side effect of always initializing its global variables, even when no symbol can be imported or none of the imported symbols are used.
The compiler will optimize out any unneeded global variable, but only importing the modules one actually needs is to be preferred.

Circular dependencies between modules are allowed, but global variables cannot create dependency loops. This is explained in more detail in Global initialization order
